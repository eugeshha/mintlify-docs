## Зачем эта дока

Это твоя личная база знаний по DevOps:

- **Конспекты лекций** (что такое Kubernetes, из чего состоит и т.д.)
- **Домашние задания** по лекциям
- **Справочник по темам**: Linux, сети, Git, Docker, Kubernetes, CI/CD, IaC, облака, мониторинг, безопасность
- **Примеры команд и файлов из git** (манифесты, скрипты, пайплайны)

## Структура

Договоримся о простой структуре:

- **`roadmap.mdx`** — общий план обучения и чеклист тем
- **`kubernetes/lecture-01-intro.mdx`** — конспект первой лекции по Kubernetes (азы, из чего состоит)
- **`kubernetes/homework-01.mdx`** — ДЗ к первой лекции

Дальше ты сможешь добавлять:

- новые лекции по Kubernetes: `kubernetes/lecture-02-...`, `kubernetes/lecture-03-...`
- новые ДЗ: `kubernetes/homework-02.mdx`, `kubernetes/homework-03.mdx`
- отдельные страницы по темам (например, `linux.mdx`, `docker.mdx`, `git.mdx`) — когда захочешь закрепить теорию.

## Как фиксировать конспект лекции по Kubernetes

Рекомендуемый формат страницы лекции (например, `kubernetes/lecture-01-intro.mdx`):

- **Краткое описание лекции**
- **Ключевые термины** (Pod, Node, Cluster, Control Plane и т.д.)
- **Картинки / схемы** (можно класть в `logo/` или отдельную папку `images/` и вставлять через стандартное MDX-изображение)
- **Команды, которые использовал** — в код-блоках
- **Что осталось непонятным** — чтобы вернуться позже

## Где хранить домашки по лекциям

Домашку по каждой лекции удобно хранить на отдельной странице:

- `kubernetes/homework-01.mdx`
- `kubernetes/homework-02.mdx`

Внутри страницы ДЗ:

- **Формулировка задания**
- **Краткий план, как решил**
- **Команды и манифесты, которые писал**
- **Выводы / грабли**, на которые наступил

Так ты сможешь быстро вернуться к любому ДЗ и вспомнить, как ты это делал.

## Как добавлять команды и файлы из git

### Команды (kubectl, docker, git и т.д.)

Любую команду удобно добавлять в код-блок:

```bash
kubectl get pods -A
kubectl describe pod my-app-123
```

Для git:

```bash
git status
git log --oneline --graph --all
```

### Файлы из репозитория (манифесты, Dockerfile, пайплайны)

Есть два варианта:

1. **Короткий фрагмент прямо в доке:**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: nginx:1.27
```

2. **Целый файл — хранишь в репо и вставляешь важные части:**

- файл лежит в git (например, `k8s/pod-my-app.yaml`)
- в доке даёшь ссылку и ключевые куски:

```markdown
Файл: `k8s/pod-my-app.yaml`
Главное:
- образ `nginx:1.27`
- один контейнер `app`
```

### Как не утонуть в коде

- В доку добавляй **только то, что нужно для понимания** (суть манифеста, ключевые поля)
- Полные файлы храни **в git-репозитории**, а в доке — выдержки и комментарии, зачем это так сделано
